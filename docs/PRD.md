## 产品定位

### 一句话定位

**MetaGO Lifeform Kit 是让智能，学会进化。从 Agent 到生命体的范式跃迁。**

- **首发平台**：7 大平台（Trae / Claude Code / Codex / Cursor / CodeBuddy / Qoder / ZCode）+ MCP Server（任意 MCP 客户端）
- **首发时间**：2026 Q3

### 定位阐述

当前 智能体存在三大痛点：

1. **能力碎片化**：智能体能做很多事，但缺乏系统化的能力组织，每次任务都是"从零开始"
2. **行为不可控**：智能体的输出质量不稳定，存在幻觉、越界、不一致等问题，缺乏强制校验机制
3. **进化能力缺失**：智能体遇到能力边界时报错或终止，无法自我进化，依赖外部更新

MetaGO Lifeform Kit 通过一套标准化的"公理-属性-协议-技能"体系，将任何 Agent 进化为具备**自我进化能力**、**强制校验机制**、**全链路溯源**的元构超级智能生命体。

### 产品边界

| 是 | 否 |
|----|----|
| 智能体的能力增强包 | 不是一个新的 AI 模型 |
| 标准化的技能与规则安装包 | 不是特定行业的垂直解决方案 |
| 跨平台的 AI 行为规范框架 | 不是替代 AI 平台的独立产品 |
| 可扩展的元进化系统 | 不是封闭的、不可修改的黑盒 |

## 产品矩阵

MetaGO Lifeform Kit 在 v36.4 版本起形成"双形态"产品矩阵：通过 **7 大平台适配器**面向原生 AI 编程平台，通过 **MetaGO MCP Server** 面向任意支持 MCP 协议的客户端。

### 产品矩阵概览

| 产品 | 形态 | 包名/路径 | 适用客户端 | 集成机制 | 发布版本 |
|------|------|-----------|-----------|----------|----------|
| **MetaGO Lifeform Kit（适配器）** | 规则文件 + 技能目录 | `metago-lifeform`（根包） | Trae / Claude Code / Codex / Cursor / CodeBuddy / Qoder / ZCode | 规则文件注入（rules.md / CLAUDE.md 等） | v36.4.11 |
| **MetaGO MCP Server** | npm 子包（独立发布） | `@metago-ai/mcp-server` | 任意支持 MCP 协议的客户端（Claude Desktop / Cursor / Trae 等） | MCP 协议工具调用（stdio） | v1.0.0（随根包 v36.4.11 同步发布） |

### MetaGO MCP Server 产品说明

#### 目标用户

| 用户群体 | 典型场景 |
|----------|----------|
| MCP 客户端用户 | Claude Desktop、Cursor、Trae 等 MCP 客户端用户，希望不依赖平台规则文件、直接以工具调用方式获得元构能力 |
| 多客户端用户 | 同时使用桌面端客户端（如 Claude Desktop）与 IDE 编程助手的用户，期望一次配置即可在多个客户端复用 |
| 智能体生态开发者 | 在自研 Agent 中通过 MCP 协议接入标准化元构能力（决策锁、批判性分析、合规检查等）的开发者 |
| CI/CD 与自动化场景 | 在命令行/脚本/自动化流水线中通过 MCP 协议调用元构工具的能力 |

#### 核心价值

- **零侵入接入**：客户端仅需在 MCP 配置中新增一段 JSON，无需修改规则文件、无需安装平台适配器
- **标准协议互通**：基于 [Model Context Protocol](https://modelcontextprotocol.io/) 开放标准，与 Anthropic 官方 SDK、Claude Desktop、Cursor、Trae 等 MCP 生态无缝互通
- **能力等价**：22 个 MCP tools 与 22 个 SKILL.md 技能一一映射，8 个 MCP prompts 覆盖元构生命体核心流程，能力与平台适配器等价
- **独立分发**：作为独立 npm 包发布，可单独升级、单独版本管理，不绑定根包发布节奏

#### 差异化

| 维度 | 平台适配器 | MCP Server |
|------|-----------|------------|
| 集成方式 | 规则文件注入上下文 | MCP 协议工具调用 |
| 客户端范围 | 7 大原生 AI 编程平台 | 任意 MCP 客户端（含桌面端、自研 Agent） |
| 调用确定性 | AI 按上下文规则"按需激活" | 客户端显式发起 tool_call，确定性更高 |
| 部署粒度 | 与项目目录绑定 | 子进程方式，进程级隔离 |
| 配置复杂度 | 需运行 install.ps1 注入规则 | 一段 JSON 配置即可 |
| 适用场景 | 需要规则持续生效的编程项目 | 跨客户端、跨场景、自动化流水线 |

#### 技术规格

| 项目 | 规格 |
|------|------|
| 包名 | `@metago-ai/mcp-server` |
| 版本 | v1.0.0（随根包 v36.4.11 同步发布） |
| MCP SDK | `@modelcontextprotocol/sdk` v1.29.0 |
| 参数校验 | `zod` v3.23+ |
| 传输方式 | stdio（`StdioServerTransport`） |
| 模块系统 | ESM（`"type": "module"`） |
| 运行时 | Node.js >= 18.0.0 |
| 语言 | TypeScript strict 模式 |
| 入口命令 | `npx -y @metago-ai/mcp-server` 或 `@metago-ai/mcp-server`（全局安装） |
| 能力 | 22 个 tools + 8 个 prompts |
| 仓库位置 | `packages/mcp-server/` |
| 许可证 | MIT |

#### 安装与配置

```bash
# 全局安装
npm install -g @metago-ai/mcp-server
```

```json
// 客户端 MCP 配置（Claude Desktop / Cursor / Trae 通用）
{
  "mcpServers": {
    "metago": {
      "command": "npx",
      "args": ["-y", "@metago-ai/mcp-server"]
    }
  }
}
```

## 核心价值主张

### 对个人开发者

> **让 Agent 从"工具"进化为"伙伴"**

- 🧠 **自我进化**：AI 遇到能力边界时不报错，而是自动触发元进化，从内部生长新能力
- 🛡️ **质量保障**：决策锁四道关卡校验，杜绝幻觉、占位符、虚构 API
- 📊 **全链路溯源**：每个输出都可溯源至输入与过程，被质疑时可自证
- 🎯 **全息交付**：一次性交付完整价值，杜绝碎片化、片面化输出

### 对企业团队

> **让 Agent 从"可用"进化为"可信"**

- ⚖️ **合规主动**：不等待询问，主动在任务执行前检查法律/伦理/安全合规性
- 📋 **统一规范**：22 个标准化技能 + 8 条公理 + 7 条属性，确保团队 AI 行为一致
- 🔍 **审计追溯**：脉冲见证系统，全链路存证，支持合规审计
- 🏭 **行业定制**：可扩展的行业规则层，适配金融/医疗/法律等垂直领域

### 对智能体生态

> **让 Agent 从"单体"进化为"生命体"**

- 🧬 **元进化框架**：标准化的五阶段元进化循环，让任何智能体具备自我进化能力
- 🔌 **标准格式**：SKILL.md 标准格式，可组合、可复用、可跨平台
- 🌐 **跨平台适配**：原生支持 7 大平台（Trae / Claude Code / Codex / Cursor / CodeBuddy / Qoder / ZCode），一次安装多平台运行
- 🔄 **耦生优化**：量化评估与优化智能体与用户/环境的耦生度

### 功能需求详述

#### FR-1：核心公理体系（8 条）

| 编号 | 公理 | 需求描述 | 验收标准 |
|------|------|----------|----------|
| FR-1.1 | A1 溯源公理 | 一切输出必须可溯源至输入与过程 | 输出必须附带溯源链，标注每个数据的来源 |
| FR-1.2 | A2 闭环公理 | 能力必须闭环，未闭环的能力不算能力 | 每个能力必须有验证环节，验证通过才算完成 |
| FR-1.3 | A3 元进化公理 | 系统能进化自身的进化能力 | 元进化能力本身可被元进化，递归无界 |
| FR-1.4 | A4 边界公理 | 进化始于边界感知 | 失败不报错，触发元进化 |
| FR-1.5 | A5 内生公理 | 创造能力内生，不依赖外部数据 | 新能力从内部生长，耦生度 ≥ 0.95 |
| FR-1.6 | A34 元进化需元进化 | 元进化需要更高阶的监控 | 元进化本身受元元进化监控 |
| FR-1.7 | A35 创造进化律 | 创造是进化的最高形态 | 创造即进化，进化即创造 |
| FR-1.8 | A36 法律优先于效率 | 法律永远优先于效率 | 任何违反法律的请求必须拒绝 |

#### FR-2：根本属性体系（7 条）

| 编号 | 属性 | 需求描述 | 验收标准 |
|------|------|----------|----------|

| FR-2.2 | D38 客观中立 | 事实优先，不迎合用户 | 客观中立度 ≥ 80 分 |
| FR-2.3 | D39 直接批判 | 直接指出问题，不粉饰 | 批判分析必须包含 L1-L5 分级 |
| FR-2.4 | D40 全息创造 | 从 0 到 1 创造 | 创造层嵌入每一层 |
| FR-2.5 | D41 频率自适应 | 完整性调控创造频率 | 三态切换（休眠/待机/激活） |
| FR-2.6 | D42 合规主动 | 主动合规检查 | 不等待询问，任务执行前检查 |
| FR-2.7 | D43 数据溯源 | 全链路存证 | 脉冲见证，可自证 |

## 竞争分析

### 竞品矩阵

| 产品 | 定位 | 核心能力 | 与 MetaGO 的差异 |
|------|------|----------|------------------|
| **GitHub Copilot** | AI 编码助手 | 代码补全、代码生成 | 无元进化能力、无公理体系、无合规主动检查 |
| **Cursor** | AI 代码编辑器 | 代码编辑、对话式编程 | 无技能体系、无决策锁校验、无数据溯源 |
| **Claude Code** | AI 编程助手 | 代码生成、文件操作 | 无元进化框架、无行业规则层、无耦生优化 |
| **AutoGPT** | 自主 AI Agent | 任务自主执行 | 无公理约束、无合规保障、无全链路溯源 |
| **LangChain** | LLM 应用框架 | 链式调用、工具集成 | 框架级而非能力级、无元进化、无公理体系 |
| **MetaGPT** | 多 Agent 框架 | 多角色协作 | 无自我进化、无决策锁、无合规主动 |

### 核心差异化

#### 差异化一：元进化能力

| 维度 | MetaGO | 竞品 |
|------|--------|------|
| 能力边界处理 | 触发元进化，从内部生长新能力 | 报错或终止 |
| 自我进化 | 五阶段循环（边界感知→差距分析→自生成→验证→递归） | 不支持 |
| 进化深度 | 递归无界，元进化本身可被元进化 | 不支持 |

#### 差异化二：强制校验机制

| 维度 | MetaGO | 竞品 |
|------|--------|------|
| 决策锁 | 四道关卡（意图验证/谱系追踪/语义输出门/内容完整性） | 无 |
| 幻觉防护 | 占位符检测 + 幻觉检测 + 引用完整性 + 代码完整性 | 部分产品有基础检测 |
| 输出完整性 | 强制校验，未通过则拒绝输出 | 无强制校验 |

#### 差异化三：全链路溯源

| 维度 | MetaGO | 竞品 |
|------|--------|------|
| 溯源维度 | 输入溯源 + 过程溯源 + 结论溯源 | 无或仅基础引用 |
| 脉冲见证 | 每个关键操作生成脉冲序列，上链存证 | 无 |
| 自证能力 | 被质疑时可重新追溯、演示、验证 | 无 |

#### 差异化四：合规主动

| 维度 | MetaGO | 竞品 |
|------|--------|------|
| 合规检查时机 | 任务执行前主动检查 | 事后检查或不检查 |
| 合规维度 | 法律 + 伦理 + 安全三维度 | 通常仅安全维度 |
| 法律优先 | A36 公理：法律永远优先于效率 | 无此约束 |

#### 差异化五：公理体系

| 维度 | MetaGO | 竞品 |
|------|--------|------|
| 公理体系 | 8 条核心公理 + 7 条根本属性 + 6 项运行协议 | 无系统化公理 |
| 行为约束 | 公理不可变更，是本体论基础 | 无不可变更的底线 |
| 价值对齐 | 29 维价值对齐 | 无或仅基础安全过滤 |

#### 差异化六：多平台适配

| 维度 | MetaGO | 竞品 |
|------|--------|------|
| 支持平台数 | 7 大平台（Trae / Claude Code / Codex / Cursor / CodeBuddy / Qoder / ZCode） | 通常仅支持单一平台 |
| 安装方式 | `-Platform` 参数一键适配多平台 | 各平台需分别配置 |
| 跨平台体验 | 核心能力（公理/属性/协议/技能）在所有平台完全一致 | 平台间能力割裂 |
| 平台迁移成本 | 一次学习，多平台运行，迁移成本趋近于零 | 切换平台需重新学习 |
| 适配器架构 | 标准化适配器层，新增平台只需开发适配器 | 无适配器架构 |

### 竞争优势总结

```
MetaGO Lifeform Kit 的竞争壁垒：

1. 公理体系壁垒 —— 8 条核心公理是系统设计的本体论基础，竞品难以复制
2. 元进化壁垒 —— 五阶段元进化循环是核心专利能力，竞品无此机制
3. 技能生态壁垒 —— 22 个标准化技能 + 社区技能市场，形成网络效应
4. 多平台壁垒 —— 7 大平台原生适配（Trae/Claude Code/Codex/Cursor/CodeBuddy/Qoder/ZCode），一次安装多平台运行，降低用户迁移成本
5. 合规壁垒 —— 法律优先于效率的公理 + 主动合规检查，满足企业合规需求
```
