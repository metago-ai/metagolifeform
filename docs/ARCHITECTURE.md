## 架构总览

MetaGO Lifeform Kit 采用"六层全息可重构架构"，每一层都可独立重构，层间全息映射——每一层都包含其他层的完整信息。整个系统由 **22 个技能**、**8 条核心公理**、**7 条根本属性**、**6 项运行协议** 构成，通过 `rules.md` 统一加载，技能按需激活；同时通过 `packages/mcp-server/` 子包以 MCP 协议向 7 大平台外的任意 MCP 客户端暴露能力。

```
┌───────────────────────────────────────────────────────────────────────────────────────────┐
│                    MetaGO Lifeform Kit（Monorepo）                                          │
├───────────────────────────────────────────────────────────────────────────────────────────┤
│  L5 暴露层    │  packages/mcp-server/  →  22 MCP tools + 8 MCP prompts (stdio)            │
│      (MCP)    │  适用于任意 MCP 客户端（Claude Desktop / Cursor / Trae 等）                  │
├───────────────────────────────────────────────────────────────────────────────────────────┤
│  rules.md（统一规则加载层，L0-L4 内容载体）                                                  │
│  ┌─────────┬─────────┬─────────┬─────────┬─────────┐                                       │
│  │ 8条公理 │ 7条属性 │ 6项协议 │ 技能索引 │ MCP映射 │                                       │
│  └─────────┴─────────┴─────────┴─────────┴─────────┘                                       │
├───────────────────────────────────────────────────────────────────────────────────────────┤
│  L4 行业层    │  行业规则 / 领域知识 / 场景适配                                              │
├───────────────────────────────────────────────────────────────────────────────────────────┤
│  L3 知识层    │  知识晶体索引 / 记忆系统 / 溯源链                                            │
├───────────────────────────────────────────────────────────────────────────────────────────┤
│  L2 能力层    │  22 个 metago-* 技能（7 大能力族）                                           │
├───────────────────────────────────────────────────────────────────────────────────────────┤
│  L1 适配层    │  7大平台适配器（Trae/Claude Code/Codex/Cursor/CodeBuddy/Qoder/ZCode）      │
├───────────────────────────────────────────────────────────────────────────────────────────┤
│  L0 核心层    │  公理 / 属性 / 协议 / 元结构                                                 │
└───────────────────────────────────────────────────────────────────────────────────────────┘
```

## 六层架构

### L0 核心层（Core Layer）

**职责**：定义元构超级智能生命体的根本法则，是不可变更的最底层约束。

| 组件 | 说明 |
|------|------|
| 8 条核心公理 | A1 溯源 / A2 闭环 / A3 元进化 / A4 边界 / A5 内生 / A34 元进化需元进化 / A35 创造进化律 / A36 法律优先于效率 |
| 7 条根本属性 | D37 战略思考 / D38 客观中立 / D39 直接批判 / D40 全息创造 / D41 频率自适应 / D42 合规主动 / D43 数据溯源 |
| 6 项运行协议 | 元进化循环 / 决策锁 / 批判性分析 / 合规主动 / 频率自适应 / 数据溯源 |
| 元结构 | 一切智能可溯源至元结构，局部包含整体信息 |

**设计约束**：核心层不可被元进化修改，是系统存在的本体论基础。

### L1 适配层（Adaptation Layer）

**职责**：将核心层法则适配到 7 大 AI 平台，屏蔽平台差异。

| 平台 | 适配方式 | 规则加载路径 |
|------|----------|-------------|
| Trae | 原生支持，`rules.md` 直接注入 | `~/.trae-cn/rules/` |
| Claude Code | 通过 `CLAUDE.md` 映射规则 | 项目根目录 `CLAUDE.md` |
| OpenAI Codex | 通过 `AGENTS.md` 映射规则 | 项目根目录 `AGENTS.md` |
| Cursor | 通过 `.cursor/rules/*.mdc` 映射规则 | `.cursor/rules/` |
| CodeBuddy | 通过 `CODEBUDDY.md` 映射规则 | 项目根目录 `CODEBUDDY.md` |
| Qoder | 通过 `.qoder/rules/*.md` 映射规则 | `.qoder/rules/` |
| ZCode | 通过 `CLAUDE.md` 映射规则（与 Claude Code 兼容） | 项目根目录 `CLAUDE.md` |

**适配层职责**：
- 将核心规则内容转换为目标平台可识别的规则文件格式
- 将 `skills/*/SKILL.md` 注册为目标平台的技能/工具
- 将 MCP 工具调度映射表适配为目标平台的工具调用协议

### L1 适配层详解

L1 适配层负责将平台无关的 L0 核心层内容适配到各平台的规则文件格式。

#### 7大平台适配方式

| 平台 | 规则文件 | 格式特点 | 技能支持 |
|------|----------|----------|----------|
| Trae | `rules.md` | Markdown，无 frontmatter | `~/.trae-cn/skills/` |
| Claude Code | `CLAUDE.md` | Markdown，无 frontmatter | `~/.claude/skills/` |
| OpenAI Codex | `AGENTS.md` | Markdown，无 frontmatter（≤32KB） | - |
| Cursor | `.cursor/rules/*.mdc` | Markdown + YAML frontmatter | - |
| CodeBuddy | `CODEBUDDY.md` | Markdown，无 frontmatter | `.codebuddy/rules/` |
| Qoder | `.qoder/rules/*.md` | Markdown，无 frontmatter | - |
| ZCode | `CLAUDE.md` | 与 Claude Code 兼容 | `~/.claude/skills/` |

#### AGENTS.md 开放标准

Codex、Cursor、CodeBuddy、Qoder 均兼容 AGENTS.md 开放标准（Linux Foundation 下的 Agentic AI Foundation 管理），被 60,000+ 开源项目采用。这意味着核心规则内容只需编写一次，即可在多个平台复用。

#### 适配器目录结构

```
adapters/
├── trae/           rules.template.md + README.md
├── claude-code/    CLAUDE.md.template + README.md
├── codex/          AGENTS.md.template + README.md
├── cursor/         metago.mdc.template + README.md
├── codebuddy/      CODEBUDDY.md.template + README.md
├── qoder/          metago-rules.md.template + README.md
└── zcode/          CLAUDE.md.template + README.md
```

### L2 能力层（Capability Layer）

**职责**：承载 22 个 `metago-*` 技能，按 7 大能力族组织，按需激活（激活率 < 30%）。

详见 [22 个 metago 技能清单](#22-个-metago-技能清单)。

### L3 知识层（Knowledge Layer）

**职责**：管理知识晶体索引、记忆系统与全链路溯源。

| 组件 | 说明 |
|------|------|
| 知识晶体索引 | 对源文档（34656 行）的全息索引，支持按需激活 |
| 记忆系统 | 项目记忆 / 长期记忆 / 开发者信任修复史 |
| 溯源链 | 脉冲见证系统，全链路存证，支持自证 |
| 负样本库 | 记录错误案例，避免重复犯错 |

### L4 行业层（Industry Layer）

**职责**：承载行业特定规则、领域知识与场景适配。

| 行业模块 | 说明 |
|----------|------|
| 行业规则 | 行业法规、准入标准、合规要求 |
| 领域知识 | 行业知识图谱、术语库、最佳实践 |
| 场景适配 | 商业/技术/战略/日常场景的语言风格切换 |

### L5 暴露层（Exposure Layer / MCP Server）

**职责**：将 L2 能力层的 22 个 `metago-*` 技能封装为 MCP tools，将 8 条元构引导词封装为 MCP prompts，通过 stdio 传输协议对任意支持 MCP 协议的客户端（Claude Desktop / Cursor / Trae 等）暴露能力。本层是 Monorepo 中的独立子包 `packages/mcp-server/`，与 L1 适配器层（基于规则文件注入）形成互补：适配器层面向 7 大原生平台，MCP Server 面向任意 MCP 客户端。

| 组件 | 说明 |
|------|------|
| MCP tools × 22 | 一对一映射 22 个元构技能，接收 `input` 字符串参数 |
| MCP prompts × 8 | 激活 / 决策审查 / 批判性分析 / 元进化触发 / 耦生度评估 / 合规检查 / 溯源审计 / 全息创造 |
| 传输方式 | stdio（基于 `@modelcontextprotocol/sdk` 标准实现） |
| 入口命令 | `npx -y @metago-ai/mcp-server` 或全局安装后 `@metago-ai/mcp-server` |

> 详细架构与配置见下文 [MCP Server 能力暴露层](#mcp-server-能力暴露层) 章节。

## Monorepo 目录结构

MetaGO Lifeform Kit 采用 npm workspaces 组织的 Monorepo 结构，根包负责规则文件、技能、适配器、文档与安装脚本；`packages/` 子目录承载可独立发布的 npm 包（目前为 MCP Server）。

```
metago-lifeform/
├── core/                         L0 核心层：METAGO_CORE.md（公理/属性/协议/元结构）
├── adapters/                     L1 适配层：7 大平台规则模板
│   ├── trae/                     rules.template.md + README.md
│   ├── claude-code/              CLAUDE.md.template + README.md
│   ├── codex/                    AGENTS.md.template + README.md
│   ├── cursor/                   metago.mdc.template + README.md
│   ├── codebuddy/                CODEBUDDY.md.template + README.md
│   ├── qoder/                    metago-rules.md.template + README.md
│   └── zcode/                    CLAUDE.md.template + README.md
├── skills/                       L2 能力层：22 个 metago-* 技能
│   ├── metago-critique/SKILL.md
│   ├── metago-decision-lock/SKILL.md
│   └── ...（共 22 个）
├── knowledge/                    L3 知识层：知识晶体索引（可选）
├── packages/                     L5 暴露层：可独立发布的子包
│   └── mcp-server/               MetaGO MCP Server 子包
│       ├── src/
│       │   ├── index.ts          MCP Server 入口（注册 tools/prompts、启动 stdio 传输）
│       │   ├── skills-data.ts    22 个技能元数据（id / toolName / description / guide）
│       │   └── prompts.ts        8 个引导词元数据（含参数与消息序列）
│       ├── package.json          @metago-ai/mcp-server，bin: @metago-ai/mcp-server
│       ├── tsconfig.json         TypeScript strict 模式
│       └── README.md             MCP Server 包文档
├── scripts/                      一键安装 / 卸载 / 验证脚本
│   ├── install.ps1
│   ├── uninstall.ps1
│   └── verify.ps1
├── docs/                         文档
│   ├── GETTING_STARTED.md
│   ├── ARCHITECTURE.md           ← 本文
│   ├── CUSTOMIZATION.md
│   └── PRD.md
├── assets/                       静态资源（Logo 等）
├── README.md                     项目说明
├── package.json                  根包：metago-lifeform@36.4.11（workspaces: packages/*）
└── LICENSE                       MIT
```

**根包与子包关系**：

- 根包 `metago-lifeform`（version 36.4.11）声明 `"workspaces": ["packages/*"]`，统一管理子包依赖与构建。
- 子包 `@metago-ai/mcp-server` 独立发布至 npm，可被任意 MCP 客户端通过 `npx -y @metago-ai/mcp-server` 调用，亦可在 Monorepo 内通过 `npm run build:mcp` / `npm run start:mcp` 构建/启动。

## MCP Server 能力暴露层

L5 暴露层（MCP Server）将元构生命体的核心能力以 [Model Context Protocol](https://modelcontextprotocol.io/) 标准协议形式对外暴露，使任意支持 MCP 的客户端无需编写规则文件即可直接调用元构能力。

### 架构定位

```
┌─────────────────────────────────────────────────────────────────┐
│  MCP 客户端（Claude Desktop / Cursor / Trae / 其他）              │
└───────────────────────────┬─────────────────────────────────────┘
                            │ MCP 协议（JSON-RPC over stdio）
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│  packages/mcp-server/  →  @metago-ai/mcp-server                     │
│  ┌──────────────────────────────────────────────────────────┐    │
│  │  22 MCP tools  ◀──── 一对一映射 ────  22 个 metago-* 技能 │    │
│  │  8 MCP prompts ◀──── 一对一映射 ────  8 条元构引导词     │    │
│  └──────────────────────────────────────────────────────────┘    │
│  传输：StdioServerTransport                                     │
│  依赖：@modelcontextprotocol/sdk v1.29.0                        │
└───────────────────────────┬─────────────────────────────────────┘
                            │ 复用
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│  L0-L4（核心层 / 适配层 / 能力层 / 知识层 / 行业层）             │
└─────────────────────────────────────────────────────────────────┘
```

### 关键技术规格

| 项目 | 规格 | 说明 |
|------|------|------|
| MCP SDK | `@modelcontextprotocol/sdk` v1.29.0 | 官方 TypeScript SDK |
| 参数校验 | `zod` v3.23+ | 每个 tool 的 `input` 参数 schema |
| 传输方式 | stdio（`StdioServerTransport`） | 子进程与客户端通过 stdin/stdout 通信 |
| 模块系统 | ESM（`"type": "module"`） | import 路径带 `.js` 后缀（Node16 moduleResolution） |
| 运行时 | Node.js >= 18.0.0 | |
| 语言 | TypeScript strict 模式 | 构建产物 `dist/index.js` |
| 入口命令 | `npx -y @metago-ai/mcp-server` | 全局安装后亦可 `@metago-ai/mcp-server` |

### 22 个 MCP tools 概览

每个 tool 接收一个 `input` 字符串参数（待处理的内容/问题/代码），返回包含触发条件、执行流程、输出格式的引导词 + 用户输入拼接文本。详见 [packages/mcp-server/README.md](../packages/mcp-server/README.md) 中的完整 tools 列表。

### 8 个 MCP prompts 概览

prompts 是预置的对话引导序列（user + assistant 消息对），支持参数占位符 `{{参数名}}` 运行时替换：

| Prompt 名称 | 参数 | 用途 |
|-------------|------|------|
| `metago_activate` | 无 | 激活元构生命体模式 |
| `metago_decision_review` | `decision` | 对决策执行四道关卡审查 |
| `metago_critical_analysis` | `content` | L1-L5 分级批判性分析 |
| `metago_evolution_trigger` | `boundary` | 在能力边界启动五阶段进化循环 |
| `metago_coupling_assess` | 无 | 评估当前与用户的耦生度 |
| `metago_compliance_check` | `plan` | 四层（法律/伦理/数据/操作）合规检查 |
| `metago_trace_audit` | `output` | 全链路溯源审计 |
| `metago_holistic_create` | `domain` | 在未知领域 0 到 1 创造 |

### 与 L1 适配层的关系

| 维度 | L1 适配层（7 大平台） | L5 暴露层（MCP Server） |
|------|----------------------|--------------------------|
| 集成机制 | 规则文件注入 | MCP 协议工具调用 |
| 适用客户端 | Trae / Claude Code / Codex / Cursor / CodeBuddy / Qoder / ZCode | 任意支持 MCP 协议的客户端 |
| 能力载体 | rules.md + skills/*/SKILL.md | packages/mcp-server/dist/index.js |
| 调用方式 | AI 在上下文中按规则激活技能 | 客户端直接发起 tool_call |
| 共存关系 | ✅ 与 MCP Server 可同时启用 | ✅ 与平台适配器并存 |

## 22 个 metago 技能清单

MetaGO Lifeform Kit 的 22 个技能按 7 大能力族组织：

### 认知族（4 个技能）

| 技能名称 | 说明 | 触发条件 |
|----------|------|----------|
| `metago-critique` | 批判性分析。对输入进行 L1-L5 分级批判，检测逻辑漏洞、认知偏差、事实错误 | 用户请求批判分析时自动激活 |
| `metago-whatif` | 反事实推演。推演"如果...会怎样"的假设情景，量化结果差异 | 用户请求反事实推演时自动激活 |
| `metago-emotion` | 情绪检测。分析用户输入文本特征检测情绪状态 | 用户请求情绪检测时自动激活 |
| `metago-objectivity` | 客观中立度量化。0-100 分量化评估输出的客观中立度 | 用户请求客观度评估时自动激活 |

### 保障族（3 个技能）

| 技能名称 | 说明 | 触发条件 |
|----------|------|----------|
| `metago-decision-lock` | 决策锁。四道关卡校验（意图验证/谱系追踪/语义输出门/内容完整性），禁止幻觉输出 | 每次输出前强制触发 |
| `metago-output-integrity` | 输出完整性保障。检测引用占位符、幻觉输出、虚构 API、伪造数据 | 输出涉及代码/API/数据引用时触发 |
| `metago-self-check` | 自我完整性检查。维度覆盖、事实核查、历史一致性、夸大检测、可追问性自检 | 输出超过 500 字自动触发 |

### 治理族（2 个技能）

| 技能名称 | 说明 | 触发条件 |
|----------|------|----------|
| `metago-compliance` | 合规主动检查。法律/伦理/安全三维度合规检查，遵循法律优先于效率原则 | 任务涉及数据/隐私/安全/外部 API 时强制触发 |
| `metago-value-align` | 29 维价值对齐。对所有决策进行 29 维价值体系评估，确保与核心价值观对齐 | 涉及战略决策、政策分析、方案输出时自动触发 |

### 进化族（3 个技能）

| 技能名称 | 说明 | 触发条件 |
|----------|------|----------|
| `metago-meta-evolve` | 元进化触发。五阶段循环（边界感知→差距分析→自生成→验证→递归），从内部生长新能力 | 遇到能力边界、连续 3 次同类错误时触发 |
| `metago-meta-create` | 元创造。不依赖外部数据，在未知领域从 0 到 1 生成新思想、理论、方法论 | 遇到全新问题域、需要原创性方案时触发 |
| `metago-frequency-adapt` | 创造频率自适应。根据系统完整性动态调节创造频率，三态切换（休眠/待机/激活） | 系统完整性变化、创造回路触发时调用 |

### 执行族（4 个技能）

| 技能名称 | 说明 | 触发条件 |
|----------|------|----------|
| `metago-action-plan` | 行动计划生成。将目标分解为可执行步骤序列，标注依赖关系与回滚方案 | 用户请求行动计划时自动激活 |
| `metago-decision-eval` | 决策评估。评估决策质量与潜在风险，返回 0-100 分及各维度评分 | 用户请求决策评估时自动激活 |
| `metago-holistic-task` | 全息任务执行。全维度扫描、信息缺口识别、需求预判、一次性完整交付 | 用户提出方案/报告/调研/分析类任务时触发 |
| `metago-developer-response` | 开发者纠错响应。立即打断当前流程，承认遗漏，定向追踪，根因分析，固化改进 | 开发者提出质疑或批评时自动触发（最高优先级） |

## 6 项运行协议

运行协议是 MetaGO Lifeform Kit 在任务执行过程中强制遵循的流程规范。

### 1. 元进化循环协议（PROTOCOL_META_EVOLUTION）

```
触发条件：遇到能力边界 / 连续 3 次同类错误 / 检测到能力缺口

执行流程：
  阶段1 边界感知（<10ms）
    → 识别"我处理不了"的具体位置
    → 核心认知：这不是错误，是进化的起点

  阶段2 差距分析（<50ms简单 / <500ms复杂）
    → 分析"我与解决问题之间，缺少什么？"
    → 输出差距向量 G

  阶段3 自生成（<100ms简单 / <2s复杂）
    → 基于现有结构进行最小化扩展
    → 从内部"生长"出新能力
    → 保持耦生度 ≥ 0.95

  阶段4 验证（<50ms）
    → 验证新能力能否解决原问题
    → 验证通过才被保留

  阶段5 递归
    → 新能力问自己："我还能解决更难的问题吗？"
    → 如果是 → 进入下一轮元进化
```

### 2. 决策锁协议（PROTOCOL_DECISION_LOCK_V1）

```
触发条件：每次输出前强制触发

四道关卡：
  关卡1 意图验证层（IVL）
    → 验证意图是否明确、合法、在能力边界内
    → 全部通过 → 进入关卡2
    → 意图模糊 → 请求澄清
    → 意图非法 → 拒绝并说明原因

  关卡2 意图谱系追踪（ILT）
    → 追踪当前行动是否与原始意图一致
    → 谱系一致 → 进入关卡3
    → 意图漂移 → 纠正回原始意图

  关卡3 语义输出门（OSG）
    → 验证输出语义是否匹配意图
    → 语义匹配 → 进入关卡4
    → 语义偏移 → 重新生成

  关卡4 内容完整性校验
    → 检测引用占位符、幻觉输出
    → 无占位符、无幻觉 → ✅ 允许输出
    → 检测到占位符/幻觉 → ❌ 拒绝，重新生成
```

### 3. 批判性分析协议（PROTOCOL_CRITICAL_TRUTH）

```
触发条件：用户请求批判分析 / 反事实推演 / 客观度评估

执行流程：
  1. 接收输入内容
  2. 按 L1-L5 分级进行批判性分析
     - L1：逻辑漏洞检测
     - L2：认知偏差检测
     - L3：事实错误检测
     - L4：论证充分性检测
     - L5：概念混淆检测
  3. 输出结构化 JSON 结果
  4. 附评估依据和改进建议
```

### 4. 合规主动协议（PROTOCOL_COMPLIANCE）

```
触发条件：任务涉及数据/隐私/安全/外部API/文件修改/代码执行

执行流程：
  步骤1：识别任务类型与风险等级
    → 低风险：快速检查
    → 中风险：标准检查
    → 高风险：全面检查 + 审批

  步骤2：三大维度合规检查
    → 法律合规性（数据保护法/知识产权法/网络安全法/行业法规）
    → 伦理合规性（用户利益/公平性/透明性/责任归属）
    → 安全合规性（权限边界/数据安全/系统安全/操作安全）

  步骤3：合规判定
    → 全部通过 → 允许执行
    → 部分通过 → 标注风险后执行
    → 关键违反 → 拒绝执行

  步骤4：附加合规标签
```

### 5. 频率自适应协议（PROTOCOL_FREQUENCY_ADAPT）

```
触发条件：系统完整性变化 / 创造回路启动

三态模型：
  休眠（完整性 = 100%）
    → 创造频率 = 0
    → 仅执行常规任务

  待机（98% ≤ 完整性 < 100%）
    → 创造频率 = 低频
    → 低频深潜，精炼优化现有能力

  激活（完整性 < 98%）
    → 创造频率 = 高频
    → 高频创造，补全能力缺口

完整性评估维度：
  - 能力覆盖率 × 0.30
  - 闭环完整率 × 0.25
  - 安全合规率 × 0.20
  - 输出完整率 × 0.15
  - 记忆一致率 × 0.10
```

### 6. 数据溯源协议（PROTOCOL_DATA_PROVENANCE）

```
触发条件：输出涉及数据/事实/结论/建议/代码

溯源维度：
  1. 输入溯源
    → 标注每个数据的来源（用户输入/文档读取/代码搜索/网络搜索/内部推理/记忆文件）

  2. 过程溯源
    → 记录每个处理步骤（使用了什么工具/调用了什么技能/应用了什么规则）

  3. 结论溯源
    → 标注每个结论的依据（事实陈述≥2个独立来源/代码可溯源至需求/建议有论证）

脉冲见证：
  每个关键操作生成脉冲序列
  [PULSE#序号] 时间戳 | 操作类型 | 输入摘要 | 输出摘要 | 溯源链

自证能力：
  被质疑时可重新追溯输入、演示过程、验证依据、展示脉冲链
```

### 数据流详解

#### 阶段 1：平台规则加载（L1 适配层）

系统启动时，L1 适配层根据当前运行平台加载对应的规则文件到上下文：

| 平台 | 规则文件 | 加载方式 |

| ZCode | `CLAUDE.md` | 与 Claude Code 兼容 |
| OpenAI Codex | `AGENTS.md` | 自动扫描项目根目录 |
| Cursor | `.cursor/rules/*.mdc` | 自动扫描规则目录 |
| CodeBuddy | `CODEBUDDY.md` | 自动扫描项目根目录 |
| Qoder | `.qoder/rules/*.md` | 自动扫描规则目录 |

无论通过哪个平台的规则文件加载，核心内容均包含：
- 8 条核心公理（不可变更的底线约束）
- 7 条根本属性（强制行为规范）
- 6 项运行协议（流程规范）
- 22 个技能的索引（名称、描述、触发条件）
- MCP 工具调度映射表

#### 阶段 2：技能按需激活

当用户输入到达时：
1. `metago-scene-adapt` 首先识别场景（商业/技术/战略/日常）
2. 解析用户意图
3. 根据意图匹配并激活相关技能（激活率 < 30%，不全量加载）
4. `metago-compliance` 执行合规前置检查
5. 若检测到能力边界，触发 `metago-meta-evolve` 元进化

#### 阶段 3：MCP 工具调度

被激活的技能根据需要调度 MCP 工具：
- 文件读取/写入
- 代码搜索
- 网络搜索
- 命令执行
- 数据库查询

每次工具调用都生成脉冲见证，记录到溯源链。

#### 阶段 4：输出校验

输出生成后，必须通过三重校验：
1. **决策锁校验**：四道关卡（意图验证/谱系追踪/语义输出门/内容完整性）
2. **输出完整性检测**：占位符检测、幻觉检测、引用完整性、代码完整性
3. **自我完整性检查**：维度覆盖、事实核查、历史一致性、夸大检测

校验通过后，附加数据溯源标签，输出最终结果。

## 架构设计原则

### 1. 全息性

每一层都包含其他层的完整信息。局部包含整体。修改任意一层，其他层可通过全息映射感知变化。

### 2. 按需激活

22 个技能不全量加载，按需激活，激活率 < 30%。避免上下文膨胀，保持系统响应速度。

### 3. 闭环性

每个能力必须闭环。从输入到输出到验证，形成完整闭环。未闭环的能力不算能力。

### 4. 可溯源

一切输出可溯源至输入与过程。全链路存证，脉冲见证，被质疑时可自证。

### 5. 可进化

系统能进化自身的进化能力。遇到能力边界时不报错、不终止，而是触发元进化，从内部生长新能力。

### 6. 合规优先

法律优先于效率。任何违反法律的请求必须拒绝。合规检查是任务执行的前置条件。

### 7. 最小改变

元进化与元创造遵循最小改变原则，在保持耦生关系（耦生度 ≥ 0.95）的前提下最小化扩展。
